### Implementation Plan (High-level)

1. Backend API contract

   - Introduce a Pydantic response model `InterfaceSummaryOut` to formalize: id, name, port_role, effective_status, occupancy, capacity.
   - Update `/api/ports/summary/{device_id}` to return `list[InterfaceSummaryOut]`.
   - For OLTs: iterate all interfaces; for PON interfaces compute occupancy per PON port via optical resolver. For other roles, set occupancy based on link-endpoint presence (or 0 if not relevant). Capacity pulled from `PortProfile` count/max_subscribers where applicable.

2. Occupancy computation

   - Use `resolve_optical_path(ont_id)`; require it to return the terminal OLT interface (or extend it). If it only returns OLT device ID today, add a helper to trace from ONT to splitter to exact OLT PON interface deterministically. Cache within request for speed.

3. Effective status derivation

   - Derive per-interface effective_status from link/device status: if any link on that interface is DOWN → DOWN; else follow device/admin overrides; otherwise UP.

4. Traffic aggregation (Phase 2)

   - Refactor the engine to roll up ONT traffic per OLT interface. Extend WS payload accordingly. This can ship after the summary endpoint is live.

5. Frontend updates

   - Cockpits fetch and render summaries. Remove all BFS and store-based traversals. Keep a tiny utility to sum occupancy by role.
   - Port matrix draws from returned interface list; colors based on effective_status with a grayscale for occupancy=0.

6. Validation & perf
   - Unit tests for backend; minimal UI tests stubbing the API. Manual flow: creating an ONT should immediately reflect in the relevant OLT port occupancy without client computation. Profile endpoint on a medium topology.
